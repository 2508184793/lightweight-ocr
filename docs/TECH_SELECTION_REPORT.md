# OCR技术选型报告

## 1. 调研概述

本项目要求不使用大型语言模型(LLM)或基于Transformer架构的模型，因此重点调研了传统OCR解决方案：Tesseract OCR、EasyOCR和PaddleOCR。

## 2. 候选方案对比

### 2.1 Tesseract OCR

**技术架构：**
- 基于LSTM（长短期记忆网络）的OCR引擎
- 使用传统图像处理算法进行文本检测
- 支持100+语言识别

**优点：**
- 历史悠久，社区成熟
- 完全开源免费
- 支持多语言
- 轻量级，资源占用低

**缺点：**
- 对复杂场景（倾斜、模糊、光照不均）识别效果较差
- 中文识别准确率相对较低
- 配置复杂，需要训练数据
- 对低质量图片拒识率高

### 2.2 EasyOCR

**技术架构：**
- 基于CRNN（卷积循环神经网络）
- 使用CRAFT进行文本检测
- 使用PyTorch深度学习框架

**优点：**
- 使用简单，API友好
- 支持80+语言
- 无需训练即可使用

**缺点：**
- 模型体积较大（约100MB+）
- 首次运行需要下载模型
- 推理速度相对较慢
- 依赖PyTorch，环境配置复杂

### 2.3 PaddleOCR（推荐方案）

**技术架构：**
- PP-OCR系列模型（PP-OCRv3/v4）
- 文本检测：DBNet算法
- 文本识别：SVTR_LCNet网络
- 基于百度飞桨(PaddlePaddle)框架

**优点：**
- **工业级精度**：中文识别准确率业界领先
- **轻量级模型**：PP-OCRv4模型仅8.1MB
- **端到端方案**：检测+识别一体化
- **多场景适配**：支持文档、自然场景、表格等多种场景
- **完善的生态**：丰富的预训练模型和工具链
- **部署灵活**：支持服务端、移动端、边缘端部署
- **活跃的社区**：GitHub 42k+ Stars，持续更新

**缺点：**
- 依赖PaddlePaddle框架
- 对英文场景优化不如中文

## 3. 选型决策

### 3.1 选择PaddleOCR的理由

1. **性能优势**：PP-OCRv4在中文场景下识别准确率显著优于其他方案
2. **模型轻量化**：8.1MB的模型大小适合各种部署环境
3. **工业级稳定性**：百度内部大规模验证，API调用量超100亿次/年
4. **非Transformer架构**：符合项目约束条件（使用CNN+RNN架构）
5. **易于集成**：提供Python API，支持多种部署方式

## 4. 项目技术栈

- **核心OCR引擎**：PaddleOCR (PP-OCRv4)
- **图像处理**：OpenCV + Pillow
- **预处理算法**：
  - 去噪：高斯滤波、中值滤波、双边滤波
  - 二值化：Otsu自适应阈值、局部阈值
  - 倾斜校正：霍夫变换、投影法
- **性能测试**：pytest + 自定义评估指标
- **API框架**：FastAPI（可选）

## 5. 预期性能指标

| 指标 | 目标值 |
|------|--------|
| 中文识别准确率 | >95% |
| 英文识别准确率 | >90% |
| 单张图片处理时间 | <500ms (CPU) |
| 支持图像格式 | JPG, PNG, TIFF, BMP |
| 模型大小 | ~10MB |

## 6. 结论

选择PaddleOCR作为项目基础，结合自定义的图像预处理模块，可以构建一个高性能、轻量级、易集成的OCR系统。该方案完全符合"不使用LLM或Transformer"的约束条件，同时提供工业级的识别准确率。
